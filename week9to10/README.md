## 三、数学与物理原理
### 1. 镜面与折射模型
*   **反射 (Reflection)**：遵循公式 $R = I - 2(I \cdot N)N$。
*   **折射 (Refraction)**：基于斯涅尔定律（Snell's Law）：
    $$n_1 \sin \theta_1 = n_2 \sin \theta_2$$
    通过判断全反射（TIR）逻辑确保玻璃材质的真实感。
### 2. 抗锯齿 (Anti-Aliasing)
利用多重采样技术，在每个像素内进行 `spp` 次随机偏移采样：
$$PixelColor = \frac{1}{N} \sum_{i=1}^{N} RayCast(u + \Delta u_i, v + \Delta v_i)$$

## 六、实验收获与思考

这一节记录的是我在写代码、调参数、改 Bug 过程中真正动手摸索出来的经验，很多结论在写代码之前并不直观，是反复跑渲染、对比画面之后才确定下来的。

### 1. 弹射次数（max_bounces）与采样率（SPP）：在"画质"和"能跑得动"之间找平衡

最开始我把 `max_bounces` 直接拉到 8，想着弹射次数越多越真实，但实际观察后发现，在我这个只有两个球体加一个地板的简单场景里，光线大多在 2～3 次弹射内就会撞到漫反射表面而终止递归（要么进入阴影分支 `break`，要么飞出场景计入背景色），弹射次数超过 4 之后画面几乎没有肉眼可见的变化，反而白白消耗 GPU 算力、拖慢交互时的帧率。所以最终把默认值定在 3～4，滑条上限保留到 8 只是为了方便后续扩展更复杂的场景。

SPP（每像素采样数）同理：单采样（spp=1）时球体边缘和棋盘格地板交界处锯齿非常明显，尤其是镜面球反射出的地板格子边缘会一格一格地"跳"；拉到 16 时画面平滑了很多，但因为每次采样都要重新走一遍完整的递归弹射，帧率掉得很厉害。来回试了几次 1 / 4 / 8 / 16 之后，发现 4 是一个在"还能流畅拖动光源"和"边缘不再明显锯齿"之间比较合适的折中点，于是把它定为默认值。

### 2. EPS 偏移量：从"哪里都加一点"到"加多大才合适"

二次以镜面/折射/阴影为主的递归光线追踪里，几乎每一次光线"重新出发"（反射后、折射后、发阴影射线时）都需要把起点沿法线方向挪一点，否则会因为浮点误差把表面自己当成遮挡物。一开始我图省事统一用了比较随意的偏移值，结果在地板与球体的接触边缘出现了一圈奇怪的黑边或者闪烁的噪点。后来把偏移量统一收敛成 `1e-4` 这个量级，并且在球体求交、平面求交里都加上对应的下限判断（要求交点参数大于这个 EPS 而不是单纯大于 0），噪点才基本消失。这也是我第一次直观体会到"理论上对的公式"在浮点数世界里还需要工程上的小修正才能稳定工作。

### 3. 玻璃材质调参：折射率、染色与透光阴影的三轮试验

*   **折射率比值**：一开始按教材给的玻璃折射率 1.5 设置 `eta_ratio = 1.0/1.5`，渲染出来的球体已经能看到明显的图像扭曲，但总觉得"通透感"不太够。尝试把折射率调低到接近水的 1.33 之后，光线弯折幅度变小，球体看起来更像一颗透明水珠而不是玻璃球，于是又调回 1.5，确认折射率对材质"质感"的影响比想象中更直接。
*   **材质染色**：最初为了让玻璃看起来"有颜色"，给它叠加了一层粉红色滤镜 `(1.0, 0.8, 0.8)`，但在多次弹射（光线在玻璃内部反复折射/全反射）之后，颜色会被多次相乘放大，导致玻璃球边缘和内部出现不自然的偏粉色泽，反而显得"脏"。后来把颜色改成纯白 `(1.0, 1.0, 1.0)`，只靠折射本身的几何畸变来表现玻璃质感，画面反而更干净、更接近真实玻璃的观感——这让我意识到对透射材质而言，颜色叠加和能量衰减的乘法关系比想象中更容易"失控"，调参时要特别小心。
*   **透光阴影**：最早的阴影逻辑非常简单——只要阴影射线路径上有任何物体，无论是镜面球还是玻璃球，直接判定为"完全阴影"。但这样会导致玻璃球在地板上投下一片死黑的阴影，完全看不出"透光"的感觉，和直觉非常不符。为了解决这个问题，我尝试把阴影检测改成一个小循环：阴影射线如果先打到玻璃球，就只衰减一部分能见度（乘以 0.95），并把射线起点沿着光线方向继续往前推进，再检测后面还有没有别的遮挡物，直到飞出场景或者撞到不透明物体为止。这个改动让玻璃球的阴影变成了浅灰色而不是纯黑色，视觉上更接近真实世界里"光透过玻璃会变暗一点，但不会完全消失"的感觉，也是这次实验里我自己觉得最有意思的一次尝试。

### 4. 一次让我印象很深的调试经历：变量解构的"隐形 Bug"

在给阴影检测加上面提到的循环逻辑时，我图方便，在不需要用到法线、颜色、材质这几个返回值的地方写了 `shadow_t, _, _, _ = scene_intersect(...)`，结果运行起来画面里偶尔会出现一些颜色明显不对、像是"串色"了的像素点，而且这些异常点的位置还不太稳定，一开始完全摸不着头脑，怀疑过是不是折射公式写错了、EPS 又没调好。后来逐步注释掉各个分支做排查，才定位到问题出在用下划线占位接收多返回值元组这件事上——在递归层数变多、变量生命周期变复杂之后，Taichi 内核对这种"占位但不用"的写法处理得不够稳定，容易在寄存器复用时把别的变量值串进来。把所有下划线占位都改成显式命名的变量（哪怕用不上）之后，那些随机出现的异常像素就再也没出现过。这次调试让我对"看起来无关紧要的写法细节，在并行编译框架里也可能引发难以预料的问题"有了更具体的认识，也养成了之后写 Taichi 内核时尽量避免占位符、把每个返回值都显式接住的习惯。

### 5. 总结

这次实验比起单纯"把公式翻译成代码"，更多时间其实花在了"代码跑起来之后画面不对劲，该往哪个方向去试"这件事上：弹射次数和采样率的取值是在画质与帧率之间反复拉扯出来的；EPS 偏移量是在噪点和黑边之间找到的折中；玻璃的折射率、染色、阴影逻辑则是三次独立的小实验各自得到的结论；而那次变量解构的 Bug 更是让我明白，并行计算框架里一些看似无关紧要的写法习惯，背后可能牵涉到底层寄存器分配这类自己平时不会主动关注的细节。整体下来，这些"调出来的经验"比直接照着公式写完代码的收获要更扎实一些。
